## Overview

Authentication behavior differs by deployment mode:

- **Local `fastskill serve`**: no built-in JWT/API-key enforcement for HTTP routes
- **Remote registry APIs**: may require authentication depending on the registry implementation

<Info>
For local development, FastSkill HTTP endpoints are accessible without `Authorization` or `x-api-key` headers.
</Info>

## Local Server (`fastskill serve`)

When running locally, these routes do not require tokens:

- `GET /api/project`
- `GET /api/skills`
- `POST /api/search`
- `POST /api/registry/publish`
- `GET /api/registry/publish/status/:job_id`
- Claude Code v1 routes under `/v1/*`

Registry publish requests use anonymous identity on local serve:

- `uploaded_by = "anonymous"`
- publish scope = `anonymous`
- returned skill id format = `anonymous/<id>`

## Remote Registry Authentication

Remote registries can still require authentication. Use CLI auth commands for those workflows:

```bash
# Authenticate against remote registry
fastskill auth login --registry https://registry.example.com

# Check stored auth info
fastskill auth whoami --registry https://registry.example.com
```

For command details, see [auth command](/cli-reference/auth-command).

## Security Guidance

If you bind local serve to non-loopback interfaces, apply network controls:

- Prefer `--host localhost` for local-only access
- Use firewall or VPC/private network boundaries
- Put the service behind a reverse proxy when exposing externally
